Duplicate classes

A large chunk of the duplicates here (the u- utility classes) came from pasting Lumos components into this site from another project. Since the site already had classes like u-gap-2, Webflow couldn't merge them and instead created u-gap-2-2, u-gap-2-3, and so on. The result is 294 Lumos utility classes with redundant numbered copies sitting unused in the stylesheet.

Critical
Performance scores are red on every device
Lighthouse gives 34/100 desktop and 25/100 mobile. Anything under 50 is considered poor and directly affects conversion. This is the highest-impact item on the list — slow pages lose customers before they even read the content.
660 duplicate CSS classes bloating the stylesheet
366 component classes and 294 Lumos utility classes each have numbered clones (button-2, u-gap-2-2, etc.). The u- duplicates came in when Lumos components were pasted from another project. Every duplicate is dead stylesheet weight and a future maintenance trap.
3 CSS variable systems running in parallel
--old-*, --swatch--*, and --photoday-design-system---* all define the same colour tokens. 28 deprecated --old-* variables are still actively used — impossible to safely remove without a full audit. Every design change risks inconsistency.
High priority
Two utility class systems competing sitewide
Lumos (u-*, 643 classes) and Client-First (padding-*, margin-*, text-*, 1,081 classes) both handle spacing and layout. When both apply to the same element, the result is unpredictable. One system needs to be deprecated and phased out.
8 legacy components still live on pages
Footer CTA Block (26 instances), Global Styles (66), Video Custom (7), NavBar_Blog, NavBar_New_Black, and others are old components still actively rendered. They carry outdated markup and styles that conflict with the current design system. Each one is a regression risk.
6+ navigation variants — should be 2
NavBar_Global_Sticky, NavBar_Global_Sticky (Starts Light), nav-main, nav-main_component, NavBar_Blog, NavBar_New_Black, and Nav_main V2 all exist as separate components. The ideal state is one nav component with a light/dark mode prop. Every extra variant is a separate place to make the same change.
No session recording or heatmaps installed
Microsoft Clarity is free and takes 5 minutes to install via GTM. Without it, there is no way to see where users click, scroll, or drop off. Setting this up before any optimisation work means decisions will be based on data rather than guesses.
What's in good shape
✓ 28 out of 32 components score 10/10 — the Lumos component library is clean, well-scoped, and consistent
✓ SEO is 100/100 on both desktop and mobile — no issues there
✓ Accessibility scores 85/100 — above average and improvable with targeted fixes
✓ Core design system (Lumos) is the right foundation — the cleanup work is removing what doesn't belong, not rebuilding from scratch
Recommended order of work
Phase Work Effort
1 · Quick wins Install Clarity via GTM, remove 3 inactive legacy components (Academy, pd-stickers, Rich Text CTA) ~10 min
2 · Navigation Consolidate 6 nav variants down to 1 component with light/dark prop, retire old nav classes ~1 day
3 · CSS cleanup Consolidate duplicate classes (660 entries), deprecate Client-First utilities in favour of Lumos u-, unify CSS variable tokens ~3–4 days
4 · Legacy components Replace 8 active legacy components (Footer CTA Block, Global Styles duplication, Video Custom, etc.) with Lumos equivalents ~2–3 days
5 · Performance Address Core Web Vitals — image optimisation, render-blocking scripts, LCP element, reduce unused CSS from steps 3–4
Effort estimates assume a single Webflow developer working in a staging environment with Lumos as the target design system. Steps 1–2 can begin immediately without risk to production. Steps 3–5 should be done in a separate staging environment before publishing.
